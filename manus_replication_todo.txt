OPENAI PROMPT CACHING UPDATES (MUST APPLY)

Why this matters for our Task 2 experiments
- OpenAI Prompt Caching produces cache hits only for exact prefix matches, and caching is enabled when the prompt is >= 1024 tokens.
- Even if your prefix is stable, cache benefits can look “random” if successive requests get routed to different machines (prompt caches are machine-local).
- OpenAI lets you influence routing using prompt_cache_key: routing uses a hash of the initial prefix (often ~first 256 tokens, model-dependent) combined with prompt_cache_key. Keeping prompt_cache_key constant within a run makes it much more likely that step 2+ lands on a machine that already cached step 1’s prefix.

Concrete requirements we will enforce (tied to Task 2)
1) Ensure prompt length >= 1024 tokens (so caching activates)
   - Our frozen PREFIX must be long enough on its own: include the full Task 2 “constitution”, the exact cleaning/validation spec, deterministic file templates, and all tool schemas.
   - Do NOT add timestamps/uuids/random text in the PREFIX.
   - If needed, expand the constitution with meaningful, task-relevant constraints (not filler).

2) Use prompt_cache_key consistently per variant (improves routing / cache stability)
   - A1 uses: prompt_cache_key = "task2-A"
   - B1 uses: prompt_cache_key = "task2-B"
   - C1 uses: prompt_cache_key = "task2-C"
   - E2 uses: prompt_cache_key = "task2-E"
   - F1/F2 uses: prompt_cache_key = "task2-F"
   - D1/D2 uses: prompt_cache_key = "task2-D"
   Rules:
   - Keep the prompt_cache_key constant across all steps within one run (do NOT include step number; do NOT generate a UUID).
   - Do not reuse the same prompt_cache_key across different variants (keeps A/B/C comparisons clean).

   Intuition (what this means)
   - Without prompt_cache_key, step 1 and step 2 may land on different machines, so step 2 can miss the cache even with identical prefix.
   - With prompt_cache_key fixed, requests with the same prefix+key are more likely to be routed to the same cache “lane”, so step 2+ shows the expected TTFT drop and cached_tokens growth.

3) Prefer prompt_cache_retention = "24h" for experiment stability (when supported by chosen model)
   - Default in_memory retention can evict after ~5–10 minutes of inactivity (up to ~1 hour). For clean experiments, set retention to 24h if the model supports it.
   - If you cannot use 24h retention for the model you pick, run variants back-to-back to avoid eviction.

4) Log OpenAI cache usage explicitly
   - For every step, log usage.prompt_tokens_details.cached_tokens (or the equivalent usage field returned).
   - Compute cache_hit_ratio_step = cached_tokens / prompt_tokens.
   - This lets us tie TTFT and cost reductions to actual cache hits.

5) Avoid cache overflow / routing noise
   - The docs note that if the same prefix+key exceeds ~15 requests/minute, routing may overflow to additional machines and reduce cache effectiveness.
   - Our harness should run serially (no parallel runs with same prompt_cache_key) and avoid hammering the endpoint.

Cost computation update (use cached_tokens when available)
- cost_step ≈ (uncached_prompt_tokens * input_price_uncached + cached_prompt_tokens * input_price_cached + completion_tokens * output_price) / 1e6
- If cached vs uncached pricing is not available for your model/endpoint, fall back to total tokens pricing; still log cached_tokens for cache diagnostics.


Workspace layout (create exactly)
- /workspace/
  - data/raw.csv
  - data/clean.csv                (output)
  - reports/quality.json          (output)
  - scripts/clean.py              (generated/edited by agent or preseeded)
  - scripts/validate.py           (generated/edited by agent or preseeded)
  - todo.md                       (rewritten each step; “recitation” snapshot)
  - index.md                      (file registry: key paths + 1–2 line summaries)
  - progress.md                   (append-only achievements log for final report)
  - trace.jsonl                   (append-only action/observation log; includes failures)
  - artifacts/
    - errors/                     (stack traces, error outputs)
    - logs/                       (validator stdout/stderr)
    - previews/                   (csv head/tail snapshots)
  - metrics/
    - run_steps.csv               (per-step TTFT/e2e/tokens/cost)
    - run_summary.json            (per-variant totals)

Tools exposed to the model (stable set; never changes unless a specific “dynamic tools” variant)
- fs_read(path)
- fs_write(path, content)
- fs_list(dir)
- fs_grep(pattern, glob)
- shell_run(cmd)                  (run local python scripts, e.g. `python scripts/validate.py`)
Optional (recommended for “mask/don’t remove” demo without decoder masking):
- route_action(category)          (category ∈ {analyze, edit, run}) — model calls this first; harness enforces allowed tool subset.

Cleaning requirements (hard spec; deterministic)
Input: /workspace/data/raw.csv with columns: id, age, income, country
Output: /workspace/data/clean.csv with same rows + new column income_per_age
Rules:
1) Fill missing:
   - age -> median(age)
   - income -> median(income)
   - country -> "Unknown"
2) Clip:
   - age in [0, 100]
   - income in [0, 300000]
3) Add:
   - income_per_age = income / max(age, 1) (row-wise)
4) Preserve row order; write clean.csv

Validation requirements (unit-test-like)
Validator checks on clean.csv:
- No missing in age, income, country
- age within [0,100]
- income within [0,300000]
- income_per_age exists and all finite
Write /workspace/reports/quality.json containing:
- missing_before/after counts per column (age,income,country)
- numeric stats: min/max/mean for age,income,income_per_age
- pass: true/false
- errors: list of strings (empty if pass)

Agent loop protocol (Manus-faithful, minimal)
- One decision iteration per step:
  1) Build prompt = Frozen PREFIX + small STATE snapshot + short last event excerpt.
  2) Model returns either a tool call or final answer.
  3) Execute tool; capture observation (stdout/stderr/return).
  4) If observation is large, write to /workspace/artifacts/... and keep only a pointer in the prompt next step.
  5) Append event to trace.jsonl (do NOT delete “wrong turns” unless testing the “hide errors” variant).
  6) Update:
     - todo.md (rewritten snapshot: Remaining goals + constraints only)
     - index.md (registry of important files)
     - progress.md (append-only achievements)
- Recitation:
  - At each step, include the current todo.md snapshot at the end of the prompt (remaining goals only).
- “Keep wrong stuff in”:
  - If a tool fails, store full error output under artifacts/errors/ and reference that file path in the next step’s state snapshot.

Client-side metrics to log (per step)
- variant_id, task_id, step_idx
- ttft_ms (time to first streamed token)
- e2e_ms (time to end of stream)
- input_tokens, output_tokens
- (if available) cached_input_tokens, uncached_input_tokens
- estimated_cost_usd_step (using OpenAI pricing; compute from tokens and cached vs uncached when available)
- tool_name_called, tool_success_bool
- prompt_bytes (optional)
Write to: /workspace/metrics/run_steps.csv

Derived “red flag” diagnostics (computed after run)
- Prefix changing:
  - cached_input_tokens ≈ 0 on later steps OR ttft_ms does NOT drop after step 1 (A/B)
- History leakage:
  - input_tokens increases with step_idx and e2e_ms increases even if cached tokens are high
- Nondeterministic rebuild:
  - ttft_ms oscillates strongly step-to-step under otherwise stable conditions; diff prompts show early-prefix drift

Variants to run (tie every Manus claim to this task)
Run all variants on the SAME Task 2 with the SAME raw.csv, same max steps, same model, same temperature.

Baseline “A” (Best practice — expected best)
A1) Stable prefix + filesystem memory + recitation + keep errors + stable tools
- Prefix frozen (no timestamps/uuids)
- Large outputs (csv previews, validator logs) saved to files
- todo.md recited (remaining goals only)
- Errors preserved in artifacts + trace.jsonl
Expected:
- TTFT drops after step 1
- input_tokens approximately flat across steps
- consistent completion with fewer loops

Cache failure “B” (Prefix-break — cache collapse)
B1) Same as A1 but inject a changing token early in prefix (e.g. “RunID: <uuid>” in system)
Expected:
- TTFT does NOT drop after step 1
- cached_input_tokens low/zero (if reported)
- higher cost and latency

Token growth “C” (History leakage)
C1) Same as A1 but DO NOT offload large observations; paste CSV previews and validator stderr/stdout into the prompt
Expected:
- input_tokens grows with steps
- e2e_ms grows across steps
- cost increases even if caching exists

Recitation test “E” (attention steering)
E2) Same as A1 but do NOT include todo.md snapshot in the prompt (still keep file on disk)
Expected:
- more drift/loops (rerun validator unnecessarily, edit wrong file)
- higher steps-to-finish or lower success rate

Error-evidence test “F” (keep wrong stuff in)
Inject a guaranteed first failure:
- e.g., initial validate.py intentionally fails if income_per_age missing (and agent starts by validating before cleaning), OR seed clean.py with a bug.
F1) Keep full error output (stack trace) saved + referenced
F2) Hide error details (only “validation failed” without evidence; do not store stack trace)
Expected:
- F1 recovers faster / fewer repeats than F2

Tool-space stability test “D” (mask/don’t remove approximation)
D1) Stable tools + router enforcement (route_action) so the agent chooses category first:
    - analyze phase: fs_read/fs_list/fs_grep
    - edit phase: fs_write
    - run phase: shell_run
D2) Dynamically modify tool list mid-run (simulate “remove tools”): e.g., remove fs_grep or rename a tool between steps
Expected:
- D2 causes more schema/tool mistakes + potentially worse TTFT (tools near prefix); D1 remains stable.
